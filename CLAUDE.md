# Hotel Jazeera Menu App

## Stack
Next.js 16 (App Router) · JavaScript (JSX, no TypeScript) · Tailwind CSS (admin) + plain scoped CSS (public v6) · Neon Postgres · Prisma ORM · Vercel

## What This Is
**View-only** digital menu for Hotel Jazeera (Al Jazeera Hotel Galkio — Galkaio, Puntland, Somalia; the "v6" design). Customer scans a QR code → browses categories with option groups + extras + tags → optionally taps **+** to build a localStorage-only **basket** they show to a waiter. There is **NO checkout, payment, or order persistence** — nothing is ever sent to a server from the customer side. Admins manage categories, items, option groups, extras, tags, banners, and social links via a protected dashboard.

## Database Schema (Prisma)
- `AdminUser` — admin accounts with JWT auth
- `Category` — name, slug, kicker, headline (HTML w/ `<em>`), sub, coverUrl, period (`morning`/`midday`/`afternoon`/`evening`/`any`; plain `String`, not a DB enum — add new values in `validations.js` `PERIODS` + the admin dropdown, no migration)
- `MenuItem` — name, description, price, imageUrl, kcal, prepTime, pairing
- `OptionGroup` — radio-style option title per item (e.g. *Bread*); `cascade` on item delete
- `ItemOption` — one row per radio choice with `priceAdd`
- `ItemExtra` — optional checkbox add-ons with `priceAdd`
- `Tag` + `ItemTag` — curated tag catalog (slug, label, variant: `default`/`green`/`spicy`)
- `Banner` — one row per service (`morning`/`midday`/`afternoon`/`evening`; `service` is a `String @unique`, upserted via POST `/api/banners`) with tagLabel, headline, body, image, CTA, 3 meta stats (legacy; not used by the v6 public menu)
- `SocialLink` — platform/value pairs
- (Removed) `Customer`, `PaymentSession`, `Order` — the checkout/payment/order flow was dropped for the view-only menu.

## Key Architecture
- Public menu (v6): `src/app/page.jsx` (server, Prisma — `fetchCategories()` is resilient and returns `[]` on DB error) → `src/components/public/RoyalShell.jsx` (client orchestrator). RoyalShell owns all state and exposes it via `MenuContext` (`menuContext.js`); screen/card components consume `useMenu()`.
- Back-button navigation (History API): the app is a single URL, so `RoyalShell` mirrors the current view into `history.state.rh` (`{ screen, catId, detailId, depth }`) to make the device/browser back button step through in-app screens instead of leaving the page. Forward actions (`openCategory`/`openBasket`/`openDetail`) `pushState`; switching chapters via the sub-tabs `replaceState` (never piles up — one back from any chapter → home). **Every** back/close affordance calls `history.back()` (`goBack`, plus `closeDetail`; the logo's `goHome` uses `history.go(-depth)` to unwind to the root). A single `popstate` listener (`applyView`, declarative + idempotent) is the only thing that reverses a view — so the device back button and the on-screen back arrow are identical and can't desync. Do **not** convert these to real routes (it would unmount/remount screens and kill the `.screen.active` animations + shared basket state).
- Branding/assets: the source brand lockup is `public/jazeera-source.jpg` ("Aljazeera Garden" 1600² leaf-"A" + wordmark on white; `public/jazeera-source.png` is its flood-filled transparent copy). Derived transparent assets are generated by `scripts/make-jazeera-assets.mjs` (flood-fills white→transparent, crops just the **leaf-"A" emblem** above the wordmark with `sharp`): `public/jazeera-icon.png` + `src/app/icon.png` (512²) + `src/app/apple-icon.png` (180²) = the leaf-"A" emblem, which Next App Router auto-serves as the favicon/tab + apple-touch icon; `public/jazeera-logo.png` (256²) is the small mark for the `HomeScreen` footer crest and the admin `.adm-brand-mark`. The **TopBar header is the leaf icon (`.tb-logo-img` → `/jazeera-icon.png`) + the text "Hotel Jazeera"** (`.tb-logo-txt`, Fraunces) — NOT a wordmark image. (Legacy `public/royal-*.png` / `rOYAL.jpeg` remain unreferenced.) To regenerate any asset, rerun `node scripts/make-jazeera-assets.mjs`.
- Public components live in `src/components/public/`: `RoyalShell`, `HomeScreen`, `CategoryScreen`, `BasketScreen`, `DetailOverlay`, `TopBar`, `DishCard`, `RowDish`, `FeaturedCard`, plus `icons.jsx`, `MenuImg.jsx` (inline `<img>` + striped fallback + **shimmer skeleton & fade-in**: shows an animated `.img-skel` overlay until `onLoad`, then fades the `.menu-img` in; a `ref`+effect catches already-cached images so the skeleton never sticks), and `menuData.js` (tag/period/price helpers). All three screens stay mounted; the active one is toggled via the `.screen.active` class.
- The v6 CSS is plain global CSS in `src/app/globals.css`, scoped under a `.rh-app` wrapper so it never collides with the admin design system. Class names: `.shell`, `.topbar`, `.searchbar`, `.rail`/`.cat-tile`, `.featured`, `.dish`/`.dish-grid`, `.row-dish`/`.dish-list`, `.show-more`, `.foot`, `.cat-hero`/`.subtabs`/`.filters`, `.detail`/`.dt-*`, `.basket-*`, `.fly`. Palette (single brand color = logo leaf green): green `#3aa935`, green-deep `#1f7a2a`, green-soft `#e8f5e6`; neutrals white `#ffffff` / near-black ink `#14171a`. The token NAMES `--royal`/`--blue`/`--emerald` are kept in `globals.css :root` + `tailwind.config.js` (admin `.adm-*` and the v6 public menu reference them) but all now resolve to the green ramp; the royal blue `#30378f` was removed entirely. Fonts: **Fraunces** (distinctive variable display serif — used for **item names and section/heading titles**, `axes: ['opsz']` + `style: ['normal','italic']` for the `<em>` accents) + **DM Sans** (all UI/chrome **and prices**). Prices use **DM Sans with `tabular-nums lining-nums`** (upright, ~15px on `.ds-price`/`.rd-price`), **not** the display serif — Fraunces renders oldstyle figures (digits with descenders) that read as italic/uneven for numerals. Both load in `layout.jsx` as `--font-fraunces` / `--font-dmsans`; **the font variable classNames are on the `<html>` element, NOT `<body>`** — `--serif`/`--sans` are declared on `:root` and reference `var(--font-fraunces)`/`var(--font-dmsans)`, so the font vars must live on the same root element or `--serif` resolves to empty and silently falls back to DM Sans (Cormorant + Inter remain on `<html>` for `/admin/*`). `--serif` (globals.css `:root`) = Fraunces; Instrument Serif was removed from the public menu. Item names (`.ds-name` ~17px/600, `.ft-name` 28/38/46px/600) are Fraunces with italic royal `<em>` accents; cards are compact (price pinned via `.ds-foot { margin-top: auto }`, no reserved `min-height`). **Dish-card foot = price (left) + the `+` add button (right)**; the add button lives in `.ds-foot` (not floating on `.ds-thumb`) and prep-time is not shown on grid cards. Grid cards (`DishCard`) show a **2-line description excerpt** (`.ds-desc`, plain text) between the name and the foot, mirroring the `RowDish` list-item `.rd-desc`. Price text via `fmt()` (`menuData.js`) strips trailing zeros (`1.50`→`1.5`, `5.98` kept, `6` whole). Cards/buttons have `:focus-visible` rings and the `@media (prefers-reduced-motion: reduce)` block neutralizes card hover/press transforms + image zoom.
- Item names (`MenuItem.name`) are rendered via `dangerouslySetInnerHTML` in all card components (DishCard, RowDish, FeaturedCard, DetailOverlay, BasketScreen) so `<em>` tags in names render as italic — matching the v6 handoff design intent.
- Category navigation: between categories you use the `.subtabs` chips or the `.cat-pager` — solid **emerald** ‹ › side discs (white chevrons, emerald-tinted shadow) so the nav control pops off the cream page and reads distinctly from the royal `.basket-cta`. The pager is rendered at the **shell level** in `RoyalShell` (a sibling of the `.basket-cta`, not inside `.cat-hero`) and is `position:fixed`, pinned low on the viewport (`bottom: max(104px, 16vh)`, above the `.basket-cta`) so it stays visible as an overlay (`z-index:30`) while the customer scrolls the dish list. It must live at shell level (not inside `.screen`) because `.screen.active` animates with `transform`, which would otherwise become the containing block for a fixed child and make it jump on every switch. It's only shown on the category screen with `>1` category and no detail open, and only on **mobile + tablet** (hidden at ≥1100px, where the chips are used instead). Switching **wraps** at the ends and plays a direction-aware `.cat-body.slide-next/prev` animation: `RoyalShell.stepCategory(dir)` records the intended direction in `navDirRef`, which `CategoryScreen`'s slide effect reads + clears via the `consumeNavDir()` callback (exposed on `MenuContext`), falling back to the index delta for lateral chip jumps, so even first↔last wraps animate the right way. There is intentionally **no touch-swipe** gesture (removed because it conflicts with the iOS edge back-swipe). All routes go through `openCategory`, which `replaceState`s on lateral switches so the back button still returns home in one press.
- The detail "Add to basket" CTA (`.dt-cta`) is **in normal flow** as the last child of `.dt-body` (after the Quantity row) — it is not a floating/`fixed` bar. The floating "· Show waiter" button (`.basket-cta`, `position:fixed` bottom-center) is shown whenever the basket has items, except on the basket screen or while a detail overlay is open. (Note: the top-bar cart is **not** actually sticky — `.shell` has `overflow:hidden`, which breaks `position:sticky` on `.topbar`; the floating `.basket-cta` is the intended always-visible basket affordance.)
- Basket (view-only): tap **+** on any card = quick-add; the detail page builds a configured line (option + extras + notes + qty). Lines persist in localStorage under `rh6_basket`; bookmarks under `rh6_bookmarks`. Nothing is posted anywhere. The basket screen ends in a "show this to your waiter" hint — there is no checkout.
- Basket line shape: `{ uid, id, name, imageUrl, catName, optionLabel, extras: [{name, priceAdd}], notes, unitPrice, qty }`.
- The "Now serving" home section features **all categories whose `period` matches the current local time** (typically two per service — e.g. breakfast + tea in the morning), via `menuData.pickPeriodCategories`. Four time bands (`menuData.currentPeriod`): morning 5–11 · midday/noon 11–15 · afternoon 15–18 · evening 18–5. (`RoyalShell` holds `periodCategories` array + a `periodCatIds` Set for the rail "Now" dots; `pickPeriodCategory`/`periodCategory` = the first, kept for compatibility). SSR uses an evening default to avoid a hydration mismatch. Each featured block is headed by **just the category name** (no "Tonight's …") and shows **9 items** (1 `FeaturedCard` + 8 `DishCard`s), with a "Show all" button when the category has >9. Assign two categories per period in `/admin/dashboard/categories` (the form supports `period`); a period with one match renders one block, evening falls back, etc.
- Admin dashboard: `src/app/admin/dashboard/` — protected by JWT cookie. Pages: categories, menu-items, tags, banners, social-links, users (no Orders page). Admin pages use Tailwind utilities + the `.adm-*` design system in `globals.css`.
- API routes: `src/app/api/` — all use Prisma + Zod validation. (No `checkout`/`order`/`payment`/`customer`/`admin/orders` routes.) Menu-affecting mutations (categories, menu-items, tags, option-groups, item-options, item-extras — both the collection and `[id]` routes) call `revalidateMenu()` (`src/lib/revalidateMenu.js` → `revalidatePath('/')`) after a successful write so admin edits appear on the public menu **immediately**; the public page (`src/app/page.jsx`) keeps `export const revalidate = 120` only as a safety-net ISR fallback.

## Workflow Orchestration

### 1. Plan Mode Default
- Enter plan mode for ANY non-trivial task (3+ steps or architectural decisions).
- If something goes sideways, STOP and re-plan immediately — don't keep pushing.
- Use plan mode for verification steps, not just building.
- Write detailed specs upfront to reduce ambiguity.

### 2. Subagent Strategy
- Use subagents liberally to keep the main context window clean.
- Offload research, exploration, and parallel analysis to subagents.
- For complex problems, throw more compute at it via subagents.
- One task per subagent for focused execution.

### 3. Self-Improvement Loop
- After ANY correction from the user: update `.claude/tasks/lessons.md` with the pattern.
- Write rules for yourself that prevent the same mistake.
- Ruthlessly iterate on these lessons until the mistake rate drops.
- Review lessons at session start for the relevant project.

### 4. Verification Before Done
- Never mark a task complete without proving it works.
- Diff behavior between main and your changes when relevant.
- Ask yourself: "Would a staff engineer approve this?"
- Run tests, check logs, demonstrate correctness.
- Run the linter after each meaningful change (`next lint` is removed in Next 16 — run `node node_modules/eslint/bin/eslint.js src/...`); the menu's intentional inline `<img>` triggers expected `@next/next/no-img-element` warnings.
- **Design-change loop:** after any UI/CSS change, launch the dev server and test the affected pages across mobile / tablet / desktop / in-between widths; iterate (fix → retest) until 100% accurate before marking done.

### 5. Demand Elegance (Balanced)
- For non-trivial changes: pause and ask "is there a more elegant way?"
- If a fix feels hacky: "Knowing everything I know now, implement the elegant solution."
- Skip this for simple, obvious fixes — don't over-engineer.
- Challenge your own work before presenting it.

### 6. Autonomous Bug Fixing
- When given a bug report: just fix it. Don't ask for hand-holding.
- Point at logs, errors, failing tests — then resolve them.
- Zero context switching required from the user.
- Go fix failing CI tests without being told how.

## Core Principles
- **Simplicity First** — make every change as simple as possible; minimal code impact.
- **No Laziness** — find root causes; no temporary fixes; senior-developer standards.
- **Minimal Impact** — changes should only touch what's necessary; avoid introducing bugs.
- **Mobile-first** — most traffic is phone-scanned QR.

## Key Rules
- Use **Prisma** for all DB queries — never raw SQL, never `$queryRaw`.
- Validate inputs server-side with **Zod** (`src/lib/validations.js`).
- JWT auth in httpOnly cookie, protected by `middleware.js`.
- Use `next/image` where possible; the menu's inline `<img>` is intentional (with a placeholder fallback).
- Public menu (v6) uses **plain global CSS scoped under `.rh-app`** in `globals.css`. Admin pages use **Tailwind utility classes** + the `.adm-*` system. Public responsiveness lives in `globals.css` `@media` blocks at `≥760px` (tablet/shell), `≥1100px` (desktop), and `≥1400px` (large); the dish grid scales 2 → 3 → 4 columns.
- iOS mobile: any `<input>` must have `font-size ≥ 16px` or iOS Safari will zoom in.
- Item totals are calculated from `unitPrice = basePrice + selectedOption.priceAdd + sum(extras.priceAdd)`.
- update the claude.md everytime code changes. 

## Seeding
```bash
source <(grep -v '^#' .env | sed 's/^/export /') && node prisma/seed-royal.js
```
Seed script uses the `PrismaPg` adapter. It now **uploads all Pexels images to Vercel Blob first** (via `prisma/lib/uploadToBlobFromUrl.js`) and stores only blob URLs in the DB. Requires `BLOB_READ_WRITE_TOKEN` to be set.

## One-time image migration (existing DB)
```bash
source <(grep -v '^#' .env | sed 's/^/export /') && node prisma/migrate-images-to-blob.js
```
Scans all `Category`, `MenuItem`, and `Banner` rows for non-blob image URLs, downloads and re-uploads each to Vercel Blob, then updates the row in place.

## Do NOT
- Reintroduce checkout, payment, order persistence, or the `Customer`/`PaymentSession`/`Order` models — this is a view-only menu.
- Reintroduce `SubItem` — it has been replaced by `OptionGroup` + `ItemOption`.
- Add service-charge or VAT rows in the basket total (decision: dropped).
- Reintroduce `src/components/public/royalStyles.js`.
- Use Drizzle ORM.
- Use `$queryRaw` or raw SQL.
- Set `font-size < 16px` on any `<input>` (the search input and notes textarea are at 16px).
